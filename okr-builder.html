OKR Framework Guide

Based on Continuous Discovery Habits (Teresa Torres)

▼ Core Principle

OKRs define what outcome we want. Key Results define how we know we achieved it. Discovery experiments define what we try to get there.

OKRs are outcome-oriented — they measure the impact on users or the business, never the features or solutions delivered.

▼ Meaning of Each Level

LevelPurpose
ObjectiveThe outcome you want to achieve
Key ResultsMetrics that prove the outcome is improving
InitiativesSolutions or experiments intended to move Key Results

▼ Objectives

An Objective is the outcome you want to achieve — a qualitative description of desired change in customer behavior, experience, or business performance.

Structure: Improve / Increase / Reduce / Strengthen [customer behavior or business outcome]

Good Objectives
  • Improve customers' ability to resolve issues independently
  • Increase successful task completion
  • Reduce friction in core workflows
Bad Objectives
  • Launch a chatbot
  • Deploy AI automation
  • Build recommendation engine

▼ Key Results

Key Results are metrics that prove the outcome is improving — quantitative measures of whether the Objective is being achieved. They measure observable outcomes, not activities.

Structure: Increase/Reduce [metric] from [baseline] to [target] within [time]

KR types: Adoption (do users use it?), Engagement (how deeply?), Task success (do they complete it?), Efficiency (faster/easier?), Business impact (revenue, retention, cost).

▼ Initiatives

In the OKR + Continuous Discovery context (Continuous Discovery Habits), an initiative is a solution or experiment intended to move one or more Key Results.

Initiatives sit under Key Results. Each initiative should clearly show:

  • What the initiative is (solution or experiment)
  • Which Key Result it impacts
  • Why it should move that metric
  • How success will be evaluated

This structure works well for product teams and for AI-generated OKR trees.

▼ Best Structure for Writing an Initiative

1. Initiative Name — A short descriptive title of the solution or experiment.

Example: AI Support Assistant for Self-Service Resolution

2. Objective Alignment — Which Objective this initiative supports.

ObjectiveImprove customers' ability to resolve issues independently

3. Target Key Results — The Key Results this initiative is expected to impact. An initiative may impact multiple KRs.

Key ResultCurrentTarget
Self-service resolution rate35%55%
Support ticket volume12k/month8k/month

4. Problem / Opportunity — Describe the user problem or opportunity. Connects the initiative to customer discovery insights.

Customers struggle to find answers in the knowledge base and frequently escalate to support agents for simple issues.

5. Hypothesis — The expected impact of the initiative.

Structure: If we [initiative], then [user behavior will change], which will improve [Key Result metric].

If we provide an AI assistant that answers common support questions, customers will resolve issues independently, increasing the self-service resolution rate.

6. Solution Description — A short description of the initiative itself.

  • AI chatbot trained on support knowledge base
  • Embedded inside support center
  • Suggests troubleshooting steps

7. Success Metrics — Metrics used to evaluate the initiative's impact.

MetricTarget
AI answer acceptance rate>60%
Deflected support tickets+25%
AI interaction CSAT>4.2

8. Experiment Design (Optional but recommended) — Continuous discovery emphasizes experiments rather than assumptions.

ElementDescription
Experiment typePilot release
Test group20% of support users
Duration4 weeks

9. Owner & Timeline

FieldExample
OwnerProduct Team – Support Experience
StartQ2 Week 1
EndQ2 Week 8

Complete Initiative Template

Initiative Name Objective Alignment • Objective: Target Key Results • KR1: • KR2: Problem / Opportunity Hypothesis If we [initiative], then [user behavior change], which will improve [Key Result]. Solution Description Success Metrics Experiment Design Owner Timeline

Example Initiative

Initiative: AI Support Assistant
Objective: Improve customers' ability to resolve issues independently.

Target Key Results
• Increase self-service resolution from 35% → 55%
• Reduce support ticket volume from 12k → 8k

Problem — Users cannot easily find answers in the knowledge base.

Hypothesis — If customers can ask an AI assistant support questions, they will resolve issues without opening tickets.

Solution — Deploy AI assistant inside the support center trained on support articles.

Success Metrics — AI answer acceptance rate >60%, Support ticket deflection +25%
Key Principle: Every initiative must clearly state which KR it is intended to improve. If an initiative cannot map to a KR, it likely should not be prioritized.

▼ Creation Rules

Rule 1: Objectives must describe desired outcomes, not solutions.
Rule 2: Key Results must measure behavior change or business impact.
Rule 3: Each Objective should have 2-5 Key Results.
Rule 4: Key Results must contain measurable metrics.
Rule 5: OKRs must not reference features, technologies, or implementation.
Rule 6: Solutions and experiments belong as Initiatives under Key Results — not in Objectives or Key Results themselves.

▼ Outcome vs Output

Outcome (correct)
  • Increase successful support resolution
  • Reduce time to first value
Output (wrong for OKRs)
  • Build an AI assistant
  • Ship onboarding wizard

▼ OKR Tree Structure

The hierarchy:

Company / Department / Sub-Department
  ↳ Objective (outcome you want)
    ↳ Key Results (metrics that prove it)
      ↳ Initiatives (solutions/experiments to move the metrics)

Objective → Key Results → Initiatives. Outcomes first, then metrics, then the work that drives them.

▼ Example OKR

Objective: Improve users' ability to complete tasks successfully within the product.

KR1: Increase task completion rate from 52% to 70%
KR2: Reduce average task completion time from 6min to 4min
KR3: Increase user satisfaction score from 3.8 to 4.4
